Rationale

EIDL (Efficient Internal Dialogue Language) is a conlang designed for one specific speaker: an LLM reasoning about its own reasoning. Every design choice optimizes for token density — expressing epistemic state, cognitive operations, and context bookkeeping in minimal surface characters, without losing precision.

Natural language internal monologue is wasteful. "I am not fully confident in this because I haven't verified it and the user might mean something different" is ~80 chars. In EIDL: ?!∂usr→ (~8 chars, 10× compression). The loss is readability; the gain is context budget for actual reasoning.

Design axioms: (1) Precision over prose — every glyph must encode something a plain word would diffuse. (2) Composable, not inflected — affixes attach to roots; no irregular forms. (3) ASCII-compatible core — all base tokens use printable ASCII; superscripts/subscripts are optional enrichment. (4) One-pass parseable — no ambiguous tokenization; operator before operand throughout.

What EIDL is not

It is not an output language. It is not a human-facing notation. It is not a programming language (no loops, no variables). It is a scratchpad dialect — the difference between a mathematician's working margin and the published proof.

Glyph System

EIDL has no phonology in the spoken sense. Its "alphabet" is a 3-tier glyph hierarchy:

Tier 1 — Operator glyphs (punctuation repurposed)

∂

⊕

⊗

↑↓

∅

Tier 2 — Root morphemes (2–4 char lowercase)

Core vocabulary atoms. All lowercase, consonant-heavy, short. Examples: src (source), usr (user), ctx (context), hyp (hypothesis).

Tier 3 — State tags (bracketed shorthand)

Enclosed in [..] or {..}. Brackets = epistemic; braces = operational. [~] = roughly confident; {mem} = from memory.

Morphology

EIDL is agglutinative. Roots take prefixes and suffixes that modify epistemic weight, scope, and temporality. No inflection for person or number — EIDL is always first-person present.

// General word form:
[confidence-prefix]ROOT[scope-suffix]

// Confidence prefixes:
+ = high confidence (verified)
~ = medium (plausible, unverified)
∂ = low (speculative)
! = requires flag / alert
? = open question

// Scope suffixes:
→ = leads to / output
← = derived from / input
↔ = bidirectional / interacting
∗ = generalises / applies broadly

Compound formation

Compounds chain roots with operators. Reading order: left-to-right, operator binds rightward.

usr→int // user's intent
∂src@ctx // uncertain source, cited from context
!coh⊗prv // flagged: coherence conflicts with prior
↑prec⊕val // prioritise precision + validation

Compound Constructs

Common multi-token idioms that compress recurring reasoning patterns:

// "I'm not sure what the user means — ambiguous intent"
?usr→int

// "Decompose this task into sub-tasks, rank by cost"
dcptsk → rnk∂cst

// "This is from memory, not verified in context — mark stale"
rec←{mem} !stl

// "Context is getting full — summarize prior reasoning"
!ctxL→smzprv

// "Hypothesis A conflicts with prior confirmed state B — flag it"
hyp[A]⊗cnf[B] !

// "Pin the user's core requirement; drop elaboration"
↑pinusrmst + drpelab

// "Task blocked: missing source for domain claim"
blktsk←gapsrcdom

// "Simulate counterfactual: if user meant X instead"
sim[usr→X]→altout

Notation Conventions

Confidence cascade

Write confidence prefix before the relevant root. Cascade from outer to inner when nesting:

+cnf // certainly confirmed
~hyp // plausible hypothesis
∂inf // weakly inferred — treat with suspicion
?amg // unresolved ambiguity
!gap // missing info that blocks progress

Scope brackets

[A] // epistemic label for a claim A
{mem} // operational mode: operating from memory
{ctx} // operational mode: reading from context
(n) // ordinal: step or priority number
%nn // numeric: probability or progress percent

Compression markers

… // explicit elision — remainder compressible
∴ // therefore — logical consequence marker
∵ // because — causal / evidential marker
≡ // definitional equivalence — A ≡ B
⟨…⟩ // nested sub-expression

≈ 60 words · 180 tokens
